## Server-Side Entities (signaling server, in-memory only)

### Room

Exists only in the signaling server's memory while peers are connecting. Deleted once both peers have established a direct P2P connection (or when the last peer disconnects).

| Field | Type | Description |
|-------|------|-------------|
| `id` | `string` | 6-character alphanumeric room code (e.g., `"ABC123"`). Generated by the server on `create_room`; returned to the host in `room_created`. |
| `peers` | `Set<WebSocket>` | Up to 2 WebSocket connections. First is host (impolite), second is guest (polite). |
| `createdAt` | `number` | Unix timestamp (ms). Used by the stale-room sweep to expire rooms with no joiner after 5 minutes. |

**State transitions:**
```
EMPTY (0 peers) → WAITING (1 peer = host) → FULL (2 peers, signaling in progress) → DELETED
```

Rooms transition to DELETED when:
- Both peers disconnect.
- The room has been WAITING for more than 5 minutes with no second peer.
- A sweep finds all sockets are closed.

### ClientRoomIndex

Reverse index: `Map<WebSocket, string>` — maps each WebSocket connection to its room ID for O(1) cleanup on disconnect. Not a domain entity; an internal data structure.

## Client-Side Entities (browser, not persisted)

### MultiplayerSession

Represents the current multiplayer session from one client's perspective.

| Field | Type | Description |
|-------|------|-------------|
| `roomCode` | `string` | The shared room code. |
| `role` | `'host' \| 'guest'` | Host runs the authoritative simulation; guest is the renderer. |
| `isPolite` | `boolean` | WebRTC Perfect Negotiation role. Guest is always polite. |
| `signalingState` | `'disconnected' \| 'connecting' \| 'signaling' \| 'connected' \| 'failed'` | Current state of the connection process. |
| `peerConnection` | `RTCPeerConnection \| null` | The underlying WebRTC connection object. |
| `gameStateChannel` | `RTCDataChannel \| null` | Unreliable/unordered DataChannel for state snapshots (host→guest). |
| `inputChannel` | `RTCDataChannel \| null` | Reliable/ordered DataChannel for player inputs (guest→host). |

### GameStateSnapshot (transmitted over `gameStateChannel`)

Binary-encoded message sent from host to guest at 20 Hz (every 50 ms).

| Field | Encoding | Description |
|-------|----------|-------------|
| `tick` | `uint32` | Monotonically increasing simulation tick counter. Guest uses this to detect out-of-order snapshots. |
| `isFullSync` | `uint8` | 1 = full snapshot of all particles; 0 = delta (only changed particles). |
| `particleCount` | `uint16` | Number of particle records in this message. |
| `particles` | `ParticleRecord[]` | Array of compact particle records (see below). |
| `p0Hp` | `float32` | Player 0 base HP. |
| `p1Hp` | `float32` | Player 1 base HP. |
| `p0Gold` | `float32` | Player 0 current gold. |
| `p1Gold` | `float32` | Player 1 current gold. |
| `p0UpgradeLevels` | `uint8[8]` | Player 0 upgrade levels for all 8 upgrade types, in order. |
| `p1UpgradeLevels` | `uint8[8]` | Player 1 upgrade levels. |

**ParticleRecord (22 bytes each, binary Float32Array/Int16Array packing):**

| Field | Encoding | Bytes | Description |
|-------|----------|-------|-------------|
| `id` | `uint16` | 2 | Unique particle ID. Used to match delta updates to existing particles. |
| `x` | `float32` | 4 | X position in world coordinates. |
| `y` | `float32` | 4 | Y position. |
| `vx` | `float32` | 4 | X velocity. |
| `vy` | `float32` | 4 | Y velocity. |
| `hp` | `float32` | 4 | Current HP. |

Total: 22 bytes/particle. 200 particles = 4.4 KB/snapshot. At 20 Hz = ~88 KB/s upstream from host.

### PlayerInputMessage (transmitted over `inputChannel`)

JSON-encoded message sent from guest to host when the guest player takes an action.

```typescript
type PlayerInputMessage =
  | { type: 'upgrade'; upgradeType: UpgradeType }
  | { type: 'nuke' }
  | { type: 'tower_build'; towerType: 'laser' | 'slow' }
  | { type: 'tower_place' }
  | { type: 'tower_upgrade'; towerIndex: number }
  | { type: 'tower_research'; towerType: 'laser' | 'slow' };
```

Sent over the reliable/ordered DataChannel — inputs must not be lost or reordered.

## Signaling Protocol Message Types

Messages sent over WebSocket between client and signaling server, or relayed between peers.

### Client → Server

| Message type | Payload | Description |
|-------------|---------|-------------|
| `create_room` | `{}` | Sent by the host to request a new room. Server generates a unique 6-char code and replies with `room_created`. |
| `join` | `{ room: string }` | Sent by the guest to join an existing room identified by the 6-char code. |

### Server → Client

| Message type | Payload | Description |
|-------------|---------|-------------|
| `room_created` | `{ room: string }` | Sent to the host after `create_room`. Contains the server-generated room code to share with the guest. |
| `joined` | `{ polite: true }` | Sent to the **guest only** after a successful `join`. Always `polite: true`; the host is always impolite. |
| `peer_joined` | `{}` | Sent to the waiting host when a guest joins. Triggers offer creation. |
| `room_full` | `{}` | Sent when attempting to join a room that already has 2 peers. |
| `room_not_found` | `{}` | Sent when attempting to join a room code that doesn't exist. |
| `peer_left` | `{}` | Sent to the surviving peer when the other disconnects. |
| `error` | `{ message: string }` | Generic server-side error. |

### Client ↔ Client (relayed verbatim by server)

| Message type | Payload | Description |
|-------------|---------|-------------|
| `offer` | `{ sdp: RTCSessionDescriptionInit }` | WebRTC offer from impolite peer. |
| `answer` | `{ sdp: RTCSessionDescriptionInit }` | WebRTC answer from polite peer. |
| `ice` | `{ candidate: RTCIceCandidateInit }` | Trickle ICE candidate from either peer. |

## State Machines

### Connection State Machine (per client)

```
IDLE ──[create/join room]──→ SIGNALING_WS
                                  │
                     [WS connected, room joined]
                                  ↓
                          WAITING_FOR_PEER  ←── (host waits here)
                                  │
                        [peer_joined received]
                                  ↓
                          NEGOTIATING  ←── (offer/answer/ICE exchange)
                                  │
              [ICE connected, DataChannels open]
                                  ↓
                              CONNECTED ──[game starts]──→ IN_GAME
                                  │
                        [peer_left or timeout]
                                  ↓
                           DISCONNECTED
```

### Room Lifecycle (server)

```
EMPTY ──[first join]──→ WAITING ──[second join]──→ FULL
                            │                         │
                      [timeout 5min]           [both disconnect]
                            ↓                         ↓
                          DELETED               DELETED
```
